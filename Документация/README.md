## 🎯 Как пользоваться документацией

### Для новых разработчиков

1. **Начните с** [Технической спецификации](Спецификации/ТЕХНИЧЕСКАЯ%20СПЕЦИФИКАЦИЯ%20КОНФИГУРАЦИИ%20PTM%20(Public%20Trade%20Module).xml) — изучите архитектуру системы
2. **Прочитайте** [Алгоритм разработки](Спецификации/Алгоритм%20разработки.xml) — поймите процесс работы
3. **Изучите** [Стандарты кодирования](Технические_стандарты/1c-standards-8.3.27.md) — научитесь писать правильный код

### При создании нового объекта

1. Используйте [Стандарт описания объектов](Спецификации/Стандарт_описания_объектов.xml) как шаблон
2. Следуйте [Алгоритму разработки](Спецификации/Алгоритм%20разработки.xml): документирование → кодирование
3. Обновите документацию согласно [Регламенту актуализации](Спецификации/Регламент%20актуализации%20документации.xml)

### При работе с формами

1. Откройте [form-elements.md](Технические_стандарты/form-elements.md) — справочник всех элементов
2. Проверьте [xml-structure-8.3.27.md](Технические_стандарты/xml-structure-8.3.27.md) для правильной структуры XML
3. Используйте [1c-standards-8.3.27.md](Технические_стандарты/1c-standards-8.3.27.md) для написания обработчиков

### При отладке

1. Настройте окружение по [testing-and-debug-setup.md](Технические_стандарты/testing-and-debug-setup.md)
2. Используйте [syntax-8.3.27.md](Технические_стандарты/syntax-8.3.27.md) для справки по синтаксису

## 📌 Важные правила

### ⚠️ Критические требования

1. **Никогда не пишите код без обновления спецификации**  
   Сначала документируем → потом кодируем

2. **Всегда используйте асинхронные методы**  
   Асинх/Ждать вместо модальных вызовов

3. **Проверяйте имена объектов через MCP-инструменты**  
   Не угадывайте — используйте `list_metadata_objects` и `get_metadata_structure`

4. **Обновляйте History после каждого изменения**  
   Дата + описание изменения

### ✅ Рекомендации

- Используйте русский CamelCase для именования
- Размещайте код в стандартных областях (#Область)
- Избегайте запросов в циклах
- Документируйте экспортные методы
- Следуйте принципу DRY (Don't Repeat Yourself)
